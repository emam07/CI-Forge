# CIForge — MVP-1 Spec

> Scoped to one wedge: **the PR-time CI regression comment.** Everything else (dashboards, cost attribution by author, multi-repo views, finance exports, billing, auth) is explicitly out of scope and deferred to v2.
>
> Companion to `THESIS.md`. If `THESIS.md` is the "why," this is the "what we are building first."

---

## 1. Goal

Ship a GitHub App that, on every pull request, posts a single comment of the form:

> ⚠ **CI Δ +3m12s (+22%)** vs `main` baseline (median of last 20 runs)
> - `test-integration` shard 4: **+1m40s** — likely cause: new dep `selenium-stealth` (added in `package-lock.json`)
> - `web/Dockerfile` build: **+58s** — likely cause: cache miss starting at line 14 (`ARG BUILD_SHA` change invalidates layers below)
> - `lint` ran on a docs-only change — last 100 docs-only PRs triggered it (consider a `paths-ignore`)
>
> _Estimated cost impact if merged: +$87/mo at this repo's current push frequency._

Nothing else. No dashboard, no PDF, no executive summary, no static-rule report, no auth wall. The comment **is** the product.

## 2. Non-goals (deferred to v2)

- Multi-repo dashboard
- Cost attribution by author / team
- Finance/chargeback CSV exports
- Manual file upload UI
- Static Dockerfile/workflow lint report (we'll surface static causes inline in the PR comment, not as a separate report)
- Auth, multi-tenant org management, RBAC
- Billing, plans, paywall
- PDF / Markdown export
- Slack notifications
- Self-hosted runner support beyond what GitHub's API exposes
- Custom rule configuration
- macOS / Windows runner specifics (Linux only for MVP-1)

## 3. The artifact (exact format)

The PR comment has four blocks. Order matters — most-recent-first.

**Block A — Headline (1 line):**
`⚠ CI Δ {sign}{Δ_human} ({Δ_pct}%) vs {baseline_branch} baseline (median of last {N} runs)`

If Δ < threshold (default: ≥ 30s AND ≥ 10%): swap ⚠ for ✅ and headline reads `CI Δ within noise`.

**Block B — Top 3 attributed contributors (bulleted):**
For each: `{job_name}{shard?}: {sign}{Δ_human} — likely cause: {hypothesis}`

Hypothesis is generated by a small rule pack (see §7). If we cannot produce a hypothesis, we say `cause unknown — see run logs`.

**Block C — Static observations (only if relevant, max 2):**
e.g. workflow fired on a docs-only PR; runner mismatch (a Linux-only job pinned to `ubuntu-latest` while another in the same workflow uses `macos-latest`).

**Block D — Cost line (1 line, italic):**
`Estimated cost impact if merged: {sign}${$_amount}/mo at this repo's current push frequency.`

Hidden if we don't yet have a defensible push-frequency baseline (< 7 days of history).

The comment is **edited in place** on subsequent pushes to the same PR (single comment, never a thread).

## 4. User journey

1. User installs the CIForge GitHub App on their org / repo.
2. We backfill the last 30 days of `workflow_run` + `workflow_job` history for selected repos.
3. User opens a PR. Their CI runs as normal.
4. When all runs for the PR's head SHA complete (or after a configurable timeout), we compute Δ vs baseline and post the comment.
5. On force-push / new commits to the PR, the comment is recomputed and edited.
6. On merge, the run becomes part of the baseline going forward.

No dashboard, no login, no settings page in MVP-1. Configuration is via a `.ciforge.yml` in the repo (optional; sensible defaults).

## 5. System architecture

```
GitHub ──webhook──▶ Ingest API ──▶ Postgres
                       │
                       ▼
                 Worker (queue)
                       │
                       ▼
              Regression Engine ──▶ GitHub API (post/edit comment)
```

- **GitHub App** — receives `pull_request`, `workflow_run`, `workflow_job`, `check_suite` webhooks.
- **Ingest API** — Next.js route handler, validates webhook signature, enqueues a job.
- **Worker** — processes the queue; pulls full run + job + step timings from GitHub API; writes to Postgres.
- **Regression Engine** — given a PR head SHA, fetches baseline window, computes Δ, runs attribution and hypothesis rules, formats Markdown comment, calls GitHub API to post or edit.

Single Next.js app + one worker process. Postgres for everything (no Redis, no separate analytics DB in MVP-1).

## 6. Data model (minimum)

```sql
installation (id, github_installation_id, created_at)
repo (id, installation_id, github_repo_id, full_name, default_branch)
workflow_run (id, repo_id, github_id, head_sha, head_branch, pr_number,
              event, status, conclusion, started_at, completed_at,
              total_duration_s, runner_label)
workflow_job (id, run_id, github_id, name, matrix_key,
              started_at, completed_at, duration_s, conclusion, runner_label)
workflow_step (id, job_id, name, number, duration_s, conclusion)
pr_comment (id, repo_id, pr_number, github_comment_id, last_head_sha,
            posted_at, updated_at)
```

`matrix_key` is a hash of the matrix params so we can compare shard-to-shard across runs.

## 7. Core algorithm

### 7.1 Baseline

For a PR targeting branch `B`:
- Take the last `N=20` *successful* runs of the same workflow on branch `B` (configurable; min 5 to compute).
- For each `(workflow, job_name, matrix_key)` tuple, compute median duration. Median, not mean — flaky outliers must not move it.
- If fewer than 5 baseline runs exist: post a comment with `Δ unavailable (baseline still warming up — N/5 runs collected)`.

### 7.2 Δ computation

For the PR's head SHA, for each `(workflow, job_name, matrix_key)`:
- `Δ_job = pr_duration - baseline_median`
- Roll up to workflow Δ and overall Δ.
- Filter: only surface jobs where `|Δ_job| ≥ max(20s, 5% of baseline)`.

### 7.3 Attribution & hypothesis rules

For each job in the top 3 by `|Δ_job|`, run hypothesis rules in order. First match wins.

| Rule | Detects | Hypothesis text |
|---|---|---|
| `dep-change` | `package-lock.json` / `yarn.lock` / `pnpm-lock.yaml` / `Cargo.lock` / `go.sum` / `poetry.lock` changed in PR; setup/install step duration ↑ | `new/changed dep in {lockfile}` |
| `docker-cache-miss` | Docker build step duration ↑ ≥ 30s; PR diff includes a `Dockerfile` line above an existing high-churn line; or `ARG`/`ENV` change | `cache miss starting at line {N} of {Dockerfile}` |
| `new-test-files` | Test job duration ↑; PR adds files matching test glob | `{K} new test files added` |
| `runner-change` | Job's `runs-on` label changed | `runner switched: {old} → {new}` |
| `matrix-expansion` | New matrix combinations in PR | `matrix expanded: +{K} combinations` |
| `setup-bloat` | Setup steps grew but work steps did not | `setup overhead increased ({Δ}s)` |
| _(fallback)_ | none of the above | `cause unknown — see run logs` |

Rules live in `src/rules/*.ts`, one file per rule. Each rule returns `{ matched: bool, text: string, evidence: string }`.

### 7.4 Static observations (Block C)

Two static checks for MVP-1, only:
- **Docs-only PR triggered heavy workflow**: PR diff is 100% under `docs/`, `*.md`, `README*`, but a workflow without `paths-ignore` ran. Suggest `paths-ignore`.
- **Runner mismatch**: same workflow has jobs on `macos-*` and `ubuntu-*` where the macOS job's named steps are platform-agnostic (no `xcodebuild`, no `pod`, no `swift`).

That's it. No 8-rule Dockerfile lint pack. The static analyzer is *only* in service of the PR comment, not a standalone report.

### 7.5 Cost line (Block D)

```
push_frequency = count(runs on default branch in last 30d) / 30
estimated_monthly_delta = Δ_total_seconds × push_frequency × 30 × $_per_minute / 60
```

`$_per_minute` defaults to GitHub's published Linux rate ($0.008/min). User can override in `.ciforge.yml`. Block hidden if `push_frequency < 1/day` or history < 7d.

Always say "estimated"; never "you will save."

## 8. Edge cases

- **First PR on a fresh install**: backfill historical runs first. If still no baseline: warming-up message.
- **PR retargeted** (base branch changed): recompute baseline against new base.
- **Force-push**: re-run on new head SHA, edit the existing comment in place.
- **Run still in progress**: comment is posted only after all blocking checks complete; if a run is still pending after 60 min, post partial Δ with "(partial — N jobs still running)" suffix.
- **Job renamed between PR and base**: match by step structure when possible; if no match, omit from attribution.
- **Flaky baseline**: median across 20 runs handles single-run outliers; if `IQR / median > 0.5`, mark baseline as `noisy` and widen the surfacing threshold to 60s / 15%.
- **Monorepo**: workflows scoped via `paths` filters baseline naturally; no special handling in MVP-1.
- **Re-runs**: GitHub's "re-run failed jobs" produces a new attempt; we use the latest attempt for both baseline and PR.

## 9. GitHub App permissions (least privilege)

- **Repository**:
  - `Actions` — read (workflow runs, jobs, logs metadata)
  - `Contents` — read (PR diff, lockfiles, Dockerfile contents)
  - `Pull requests` — write (post/edit comments)
  - `Metadata` — read (default)
- **Webhooks subscribed**: `pull_request`, `workflow_run`, `workflow_job`, `check_suite`, `installation`
- **Org**: none.

No `Administration`, no `Secrets`, no write access outside PR comments.

## 10. Stack

- **Runtime**: Node.js 20+
- **Web/API**: Next.js 15 (App Router) — single deployable
- **DB**: Postgres 16 + Prisma
- **Queue**: Postgres-backed (`pg-boss` or `graphile-worker`) — no Redis in MVP-1
- **GitHub client**: `@octokit/app` + `@octokit/webhooks`
- **Hosting**: Fly.io or Railway (single web + single worker process)
- **Tests**: Vitest, with fixture-driven tests for the regression engine

Rationale for staying single-language and single-DB: every moving part you don't add in MVP-1 is one you don't have to maintain when you have zero users.

## 11. Build sequence

**Week 1 — Ingest + baseline**
- GitHub App registration (dev + prod)
- Webhook receiver, signature verification, queue
- Worker that hydrates `workflow_run` / `workflow_job` / `workflow_step` from API
- Backfill script (last 30d for a repo)
- Postgres schema + Prisma
- _Exit criterion_: install on a real repo (the CIForge repo itself), see runs flowing in.

**Week 2 — Regression engine + comment**
- Baseline computation (median over window, per workflow/job/matrix)
- Δ computation + thresholding
- Rule pack: `dep-change`, `docker-cache-miss`, `new-test-files`, `runner-change`, `matrix-expansion`, `setup-bloat`, fallback
- Markdown comment formatter
- Post + edit-in-place via GitHub API
- _Exit criterion_: open a deliberate-regression PR on the CIForge repo, see correct comment.

**Week 3 — Polish + ship to 3 design partners**
- Static observations (Block C: docs-only trigger, runner mismatch)
- Cost line (Block D) with `.ciforge.yml` override
- Edge cases (warming-up, force-push, partial runs)
- Install flow / landing page (one page, "Install on GitHub")
- 3 design-partner installs, iterate on comment wording

**Explicitly not in MVP-1**: dashboard, login, multi-repo view, cost-by-author, Slack, finance exports, billing.

## 12. Success criteria for MVP-1

- 3 external repos with the App installed and a CIForge comment on at least one merged PR each.
- At least 1 case where the comment correctly attributed a regression to the actual cause (verified by author).
- Median end-to-end latency from `workflow_run.completed` webhook → comment posted: < 30s.
- Zero false ⚠ headlines on PRs whose true Δ is within noise (precision > recall in MVP-1).

## 13. Open questions

1. **Comment vs Check Run.** A GitHub Check Run is more native (appears in the PR's checks panel) but less visible. A PR comment is louder but noisier. MVP-1 default: PR comment. Reconsider after design-partner feedback.
2. **What happens on draft PRs.** Currently we comment on every PR including drafts. May want to defer until ready-for-review to reduce noise.
3. **Self-hosted runners.** GitHub's `workflow_job` API exposes runner labels; we treat them as opaque. Cost line is hidden when the runner label isn't a recognized GitHub-hosted SKU. Confirm.
4. **`.ciforge.yml` schema.** Minimal for MVP-1: `dollar_per_minute_override`, `baseline_window`, `noise_threshold_seconds`. Anything else?
5. **First-install onboarding email.** Out of scope, or a 1-line "you're installed, here's what to expect on your next PR"?
6. **Telemetry.** Do we record which comments engineers reacted to / dismissed, to tune thresholds? Privacy-sensitive — defer.

---

## 14. Bottom line

This spec deliberately rejects the "comprehensive analyzer" framing of the original CIForge spec. The product is **one PR comment**, computed from observed runs, with concrete attribution. Everything else is downstream of that artifact working.

If the comment is good, the dashboard, the cost-attribution view, and the chargeback CSV are obvious follow-ons. If the comment is not good, no amount of dashboard polish saves the product.

Build the comment first.
